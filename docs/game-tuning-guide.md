# Game Tuning Guide

This document is the quick reference for changing the basic feel, pace, and look
of the prototype. Most runtime constants are in `src/main.c`,
`src/engine/world.h`, `src/engine/ground.c`, `src/engine/mountains.c`, and
`src/engine/scenepal.c`. Some visual details are baked into generated assets in
`tools/gen_assets.py` and `tools/gen_mountains.py`; after changing those, rerun
the generator and rebuild the ROM.

## Enemy Spawning, Speed, And Count

Runtime enemy objects are controlled mainly from `src/engine/world.h` and
`src/main.c`.

- `MAX_OBJECTS` in `src/engine/world.h` is the maximum number of active enemies.
  Raising it also increases stored renderer pools and object shadows, so check
  sprite/VRAM budget after changing it.
- `WORLD_SPAWN_INTERVAL` in `src/engine/world.h` is the delay between spawn
  attempts. Smaller values spawn enemies more often.
- `WORLD_APPROACH_VZ_BASE` and `WORLD_APPROACH_VZ_RAND` in
  `src/engine/world.h` control each enemy's forward speed range. Each spawned
  enemy gets `BASE + random(0..RAND)`.
- `ENEMY_SPEED_DEFAULT`, `ENEMY_SPEED_MIN`, `ENEMY_SPEED_MAX`, and
  `ENEMY_SPEED_STEP` in `src/main.c` control the percentage multiplier applied
  to enemy approach speed. Buttons A/B adjust this value at runtime.
- Enemy lateral spawn spread is in `spawnObject()` in `src/main.c`:
  `o->wx = random() % 281 - 140`.
- Enemy height variation is also in `spawnObject()`:
  `o->wy` is either ground-level `0` or a random airborne value from `20..70`.
- Enemy sideways drift is in `spawnObject()`:
  `o->vx = random() % 5 - 2`.

Enemy scale speed is a result of depth movement. To make enemies grow faster on
screen, increase `WORLD_APPROACH_VZ_BASE`, `WORLD_APPROACH_VZ_RAND`, or the
runtime enemy speed percentage. To change the projection itself, use
`WORLD_Z_NEAR`, `WORLD_Z_FAR`, `WORLD_GROUND_DEPTH`, and `OBJ_BASE_SIZE` in
`src/engine/world.h`, but these affect every projected object and should be
changed carefully.

The stored enemy renderer in `src/prototypes/stored_frames/render_stored.c`
controls visual scale buckets:

- `FRAME_COUNT` and `FRAME_SIZES` define the available enemy scale frames.
- `FRAME_MAX_SIZE` is the largest rendered enemy size.
- `FRAME_TIERS` maps frame ranges to 16, 32, and 64 px sprite definitions.

If you change enemy max size or scale frame count, update both the generated
sprite assets and these renderer constants together.

## Trees

There are two tree systems: runtime roadside trees in `src/main.c`, and tiny
trees baked into the mountain strip by `tools/gen_mountains.py`.

Runtime roadside trees:

- `MAX_TREES` in `src/main.c` is the number of logical trees cycling through
  depth.
- `TREE_VISIBLE_COUNT` is the maximum number rendered at once.
- `TREE_NEAR_SPRITES` controls how many visible trees use the near/full-height
  sprite set. The rest use far sprites via `TREE_FAR_SPRITES`.
- `TREE_Z_SPACING` spreads logical trees between `WORLD_Z_NEAR` and
  `WORLD_Z_FAR`.
- Runtime tree forward speed currently uses `GROUND_FORWARD_SPEED` in
  `updateTrees()`. Increasing `GROUND_FORWARD_SPEED` makes the ground palette
  animation and runtime trees advance faster together.
- Tree lanes are the `lanes` array in `treeLane()` in `src/main.c`.
- `TREE_FRAME_SIZES`, `TREE_FRAME_COUNT`, and `TREE_FRAME_MAX_SIZE` control the
  runtime tree scale buckets.
- `TREE_NEAR_CANVAS_H`, `TREE_FAR_CANVAS_H`, and `TREE_FRAME_CANVAS_W` control
  sprite positioning/canvas assumptions for tree frames.

Tiny mountain-strip trees:

- `SPACING` in `tools/gen_mountains.py` controls how many tiny trees are baked
  into the 512 px mountain strip.
- The `h = 6 + ...` and `rad = 2 + ...` lines in `tools/gen_mountains.py`
  control baked tree height and canopy radius.
- `BAND`, `TREE`, and `TREE_HI` in `tools/gen_mountains.py` control the source
  colours used before palette conversion.
- `MTN_TREE_START` in `src/engine/mountains.c` controls where the mountain
  strip switches from mountain parallax to tree-band parallax.

## Ground And Checkerboard

Checkerboard geometry is generated by `tools/gen_assets.py`, while runtime
motion and recolouring happen in `src/engine/ground.c`.

Forward movement:

- `GROUND_FORWARD_SPEED` in `src/engine/world.h` is passed into `GROUND_update()`
  each active frame. It drives checker palette rotation and runtime tree motion.
- The multiplier in `src/engine/ground.c`, `fwdAcc += speed * 21`, controls how
  strongly the passed speed advances the checker palette animation. Increase it
  for faster checker motion without changing object/tree world speed.
- `CELL` in `tools/gen_assets.py` changes checker cell size in the generated
  ground art. Larger cells produce a wider/slower-looking checker pattern.
- `PHASES` and `CHECKER_BASE` in `tools/gen_assets.py` must stay aligned with
  `CHECKER_ENTRIES` and `CHECKER_PAL_BASE` in `src/engine/ground.c`.

Checker colours:

- Runtime colour schemes now live in `SCENE_SCHEMES` in
  `src/engine/scenepal.c`.
- Each scheme has `checkerLight`, `checkerDark`, and six mountain colours.
- `GROUND_setCheckerColors()` in `src/engine/ground.c` sets the two checker
  driver colours and rebuilds blend tables. It does not directly write CRAM.
- `STAGE_CHECKER` in `src/engine/ground.c` is the older checker-only stage
  palette list. Runtime level swaps should use `SCENE_SCHEMES` instead.

Horizontal line scrolling and ground sway:

- `GROUND_SWAY_SHIFT` in `src/engine/ground.c` controls the strength of lateral
  sway across scanlines. Smaller shift means stronger sway; larger shift means
  gentler sway.
- `VANISH_SHIFT` controls how much player/world X affects the vanishing-point
  offset. Smaller shift means stronger vanishing-point movement.
- `HSCROLL_MIN`, `HSCROLL_MAX`, and `HSCROLL_CENTER` are clamp and centering
  values derived from board width and viewport width.
- `CHECKER_START_PAD` moves the first visible checker rows below the horizon to
  avoid near-horizon shimmer.
- The main loop currently rebuilds ground line scroll every other frame with
  `(scrollPhase & 1) == 0` in `src/main.c`. Rebuilding every frame is smoother
  but costs more DMA/CPU.

Pitch and horizon:

- `GROUND_HORIZON` in `src/engine/ground.h` is the neutral on-screen horizon.
- `PITCH_RANGE` and `BOARD_PAD_TOP` in `src/engine/ground.c` must match
  `tools/gen_assets.py`.
- `GROUND_VISIBLE_HORIZON_PAD` in `src/engine/ground.h` is used to anchor
  projected objects just below the visible checker horizon.
- `PLAYER_MIN_Y`, `PLAYER_GROUND_SINK`, `PLAYER_FEET_Y`, and
  `WORLD_GROUND_DEPTH` affect how the player, horizon, and object contact points
  line up.

## Mountains, Sky, And Clouds

Mountain strip runtime:

- `MTN_STRIP_H`, `MTN_STRIP_H_TILES`, and `MTN_STRIP_W_TILES` in
  `src/engine/mountains.c` must match the mountain image dimensions.
- Mountain parallax uses `mtnScroll = swayX >> 3`; the tree band uses
  `treeScroll = swayX >> 2`. Smaller shifts create stronger parallax.
- `MTN_PAL_FIRST_COLOR` and `MTN_PAL_COLOR_COUNT` define the six PAL0 entries
  used by the mountain strip.

Mountain art generation:

- `STRIP_W`, `STRIP_H`, and `GRAD_TOP` in `tools/gen_mountains.py` control
  mountain image size and crest limits.
- `ridge_terms`, `ridge_base`, and `ridge_amp` control the mountain silhouette.
- `GREEN_FAR`, `GREEN_NEAR`, `CREST`, `BAND`, `TREE`, and `TREE_HI` are the
  generator source colours for the original mountain asset.

Sky and clouds:

- `initSkyPalette()` in `src/main.c` sets the active PAL3 sky gradient colours.
- `gen_sky_gradient()` in `tools/gen_assets.py` controls the generated sky tile
  gradient and stop indices.
- `CLOUDS_applyPalette()` in `src/engine/clouds.c` sets PAL2 indices 12..15 for
  cloud colours.
- `gen_clouds()` in `tools/gen_assets.py` controls cloud art shape, size, and
  source palette.

## Level Palette Swaps

Level palette scheme data is centralized in `src/engine/scenepal.c`.

- Add or edit schemes in `SCENE_SCHEMES`.
- Scheme 0 should remain the boot/default look unless you intentionally want a
  new default.
- `FADE_KEYFRAMES` controls how many computed colour snapshots a fade uses.
- `FADE_HOLD` controls how many frames each snapshot is held.
- Pressing C calls `SCENEPAL_cycle()` from `handleInput()` in `src/main.c`.
- `SCENEPAL_update()` runs before `GROUND_update()` so checker driver colours
  are used by the same frame's normal checker palette upload.
- Mountain palette writes use `MOUNTAINS_setColors()` and are skipped when a
  keyframe does not change any quantized VDP colour.

Colour values use `RGB24_TO_VDPCOLOR(0xRRGGBB)`. The Mega Drive stores 3 bits
per channel, so nearby RGB24 values often quantize to the same hardware colour.

## Player, Shots, And HUD Feel

- `PLAYER_SPEED_DEFAULT` and `PLAYER_SPEED_PCTS` in `src/main.c` control player
  movement speed. Button Z cycles through the percentage table.
- `PLAYER_MIN_X`, `PLAYER_MAX_X`, `PLAYER_MIN_Y`, and computed `playerMaxY`
  define the movement box.
- `WORLD_SHOT_SPEED` in `src/engine/world.h` controls shot travel speed.
- `MAX_SHOTS` in `src/engine/world.h` controls shot pool size.
- `SHOT_SHADOW_SIZE` in `src/main.c` controls shot shadow size.
- `HURT_FLASH_FRAMES` and `HURT_FLASH_PHASE` in `src/main.c` control damage
  flicker duration and blink cadence.
- HUD output is updated through `HUD_update()` in `src/main.c`; the displayed
  object count comes from `countObjects()`.

## Sprite And Shadow Budgets

- `SPRITE_POOL_TILES` in `src/main.c` is the global sprite tile pool size passed
  to `SPR_initEx()`.
- Enemy sprite pools are built from `MAX_OBJECTS` and `FRAME_TIERS` in
  `src/prototypes/stored_frames/render_stored.c`.
- Shadow capacity is `SHADOW_REGISTRY_COUNT` in `src/engine/shadow.c`; it is
  based on `MAX_OBJECTS`, `MAX_SHOTS`, the player, and runtime trees.
- Shadow visual size buckets are controlled by `SHADOW_FRAME_SIZES`,
  `SHADOW_FRAME_COUNT`, `SHADOW_FRAME_MAX_SIZE`, and `SHADOW_MIN_SIZE` in
  `src/engine/shadow.c`.

When increasing object, tree, shot, or shadow counts, rebuild and test in an
emulator with sprite limits visible if possible. The Mega Drive sprite engine can
drop sprites on busy scanlines before CPU-side allocation fails.

## Regeneration And Build

When changing only C constants, rebuild with:

```sh
make build
```

When changing generated art in `tools/gen_assets.py` or `tools/gen_mountains.py`,
rerun the relevant generator first, then rebuild. Generated PNGs live under
`res/sprites/` and are pulled into the ROM by `res/resources.res`.
